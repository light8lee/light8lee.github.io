## 先用五句话抓住整篇论文

1. **HSCodeComp 提出一个面向“层级规则应用”的 Agent 基准，要求模型根据含噪的商品文本、属性和图片，严格按照关税规则预测唯一的美国 10 位 HS 编码。**
2. **数据集包含 632 个真实商品、32 个类别和 352 个不同编码，由 26 位关税专家通过双人独立标注、专家裁决和抽样复核构建。**
3. **对 23 个 LLM、VLM 和 Agent 的实验表明，最佳 Agent 的准确率仅为 46.8%，远低于人类专家的 95.0%，且错误会随分类层级加深而累积。**
4. **消融实验发现官方税则至关重要，图片和历史裁定可提供辅助，但简单加入专家规则、增加推理步骤、多数投票或自我反思反而可能造成“推理漂移”。**
5. **论文的核心启示是：专业 Agent 的关键不是想得更久，而是采用结构化事实抽取、可执行规则、逐层验证和可追溯证据链来约束推理。**

可以把整篇论文理解成回答五个问题：

1. 为什么普通检索和分类 Benchmark 不够？
2. 怎样构建一个可信的专家级规则应用 Benchmark？
3. 当前最强模型究竟差在哪里？
4. 哪些常见 Agent 增强方法有效，哪些反而有害？
5. 下一代专业 Agent 应该怎样设计？

## 论文定位：它提出了核心方法，还是主要在拆解问题？

### 先给结论

严格来说，HSCodeComp 的核心贡献是 **Benchmark 构建、任务建模和系统性问题诊断**，不是提出一个新的、完整的 HS 编码解决算法。

更准确的定位是：

> 论文负责证明“现有 Agent 不能稳定应用层级规则”，定义应该怎样评测这种能力，并通过实验解释系统为什么失败；它提供了数据、专家流程和诊断证据，但没有给出能够消除这些失败的最终方法。

因此，如果把论文分成“提出问题、建立测量工具、分析原因、给出解决方案”四部分，它完成得最好的是前三部分：

| 研究环节 | 论文完成程度 | 具体贡献 |
|---|---|---|
| 提出问题 | 很强 | 将层级规则应用定义为区别于网页搜索和数据库查询的 Agent 能力 |
| 建立测量工具 | 很强 | 构建真实、含噪、多模态、专家标注的 HSCodeComp |
| 分析失败原因 | 很强 | 识别推理漂移、过早决策、信息误处理、规则误用等问题 |
| 提出新解决算法 | 较弱/没有 | 没有训练新模型，也没有提出完整的规则执行或确定性 Agent 架构 |

### 论文实际包含的三类“方法”

虽然它不是方法论文，但并非完全没有方法，主要包含以下三类研究方法。

#### 方法一：任务建模方法

作者把传统的 HS Code 文本分类重新建模为：

```text
商品信息 X + 层级规则 R + 领域知识 K → 10 位编码 Y
```

这次重新建模很重要，因为它改变了对问题的理解：

- 输入不只是商品标题，还包含属性、图片、价格和电商类目；
- 中间过程不只是计算语义相似度，还必须使用官方税则；
- CROSS 历史裁定只能作为领域知识，不能替代规则；
- 输出不是任意标签，而是分类树中的一条合法路径；
- 每一层决定都受到父级规则、排除条款和同层比较约束。

换句话说，作者没有发明一个更强分类器，而是论证“把它当分类问题本身就不完整”。

#### 方法二：Benchmark 构建方法

论文设计了一套高成本专家标注流程：真实电商采样、保留现实噪声、双专家独立标注、高级专家裁决、无共识样本丢弃，以及随机 10% 独立复标。

这一部分是论文最明确的方法学贡献。它回答的是：怎样为含有模糊规则和专业争议的任务建立相对可信的评测真值。

#### 方法三：评测与诊断方法

论文没有只报告最终 10 位准确率，而是：

- 分别测量 2、4、6、8、10 位层级准确率；
- 计算条件准确率，定位错误发生在哪一层；
- 比较基础模型、视觉模型、开源 Agent 和商业 Agent；
- 移除税则、专家规则、CROSS 和图片进行消融；
- 改变推理深度，测试多数投票和自我反思；
- 结合案例建立感知、推理和反思失败分类。

这些实验共同构成了一套“如何诊断层级规则 Agent”的方法。

### 论文提供了一个基础解题范式，但不是新算法

根据专家标注过程和 Agent 工具配置，可以提炼出论文中的基础工作流：

```text
提取商品事实
→ 检索官方税则
→ 查询 CROSS 历史裁定
→ 按专家决策原则处理冲突
→ 沿 Chapter / Heading / Subheading 逐层分类
→ 验证最终编码是否存在且有效
```

不过，这只是专家操作流程和 Agent 应使用哪些资源的描述，还不能算一个完整的新算法，原因包括：

1. 每一步由谁控制、何时进入下一步，没有被设计成严格的状态机；
2. 六条专家规则仍以自然语言提示词存在，没有转成可执行约束；
3. 候选编码如何生成、比较和排除，没有统一算法；
4. 出现文本冲突、图片冲突或规则冲突时，没有可靠的冲突消解器；
5. 反思何时触发、何时停止，没有稳定机制；
6. 论文没有通过训练或架构创新解决它所观察到的推理漂移。

### 实验中的 Agent 也不是作者提出的新解法

SmolAgents、WebSailor、AWorld、AgentOrchestra 等都是已有 Agent 框架。论文主要把它们接入统一的搜索、税则和知识检索工具，并比较它们在 HSCodeComp 上的表现。

因此，最佳结果 46.83% 应理解为“已有 Agent 在该 Benchmark 上的强基线”，而不是论文提出的新模型达到的结果。

论文也没有：

- 训练一个新的 HS 分类模型；
- 提出新的 Agent 规划算法；
- 构建可执行税则规则引擎；
- 提出新的检索或重排序模型；
- 用强化学习直接优化层级规则应用；
- 给出已经达到专家水平的生产系统。

### 论文真正留下的解决方向

虽然没有完成解决方案，论文的消融和失败案例已经比较明确地指向下一步架构：

```text
结构化事实抽取
+ 证据来源和可靠度建模
+ 多候选生成
+ 可执行规则引擎
+ 逐层约束验证
+ 反证检索
+ 有触发条件的局部反思
+ 可审计证据链
```

其中最关键的变化，是从“让一个通用 Agent 自由规划并写长推理”，转向“使用确定性控制流，把 LLM 限制在局部语义判断环节”。

2026 年的后续独立工作 *A Deterministic Agentic Workflow for HS Tariff Classification* 更接近真正的方法论文：它使用固定六阶段流程，将模型调用限制在较窄的步骤，并要求阶段化结构输出和规则引用。它可以看作对 HSCodeComp 所暴露问题的一种初步回答，但不是 HSCodeComp 原论文自身提出的方法。

### 阅读时应该如何评价这篇论文

如果期待的是“一个新的 HS 分类 SOTA 算法”，这篇论文确实没有给出完整答案；如果关注的是“如何发现并测量专业 Agent 的关键能力缺口”，它的贡献很强。

最准确的一句话评价是：

> HSCodeComp 是一篇资源与诊断论文，而不是解决方案论文；它把问题定义清楚、把现有系统的失败测量清楚，也为后续方法研究划出了明确靶点。

## 第一章：为什么要评测“层级规则应用”

对应第一句话：

> HSCodeComp 提出一个面向“层级规则应用”的 Agent 基准，要求模型根据含噪的商品文本、属性和图片，严格按照关税规则预测唯一的美国 10 位 HS 编码。

## 1.1 论文观察到的评测空缺

许多 Agent Benchmark 主要测两种能力。

第一种是开放域搜索：Agent 能不能浏览网页、整合长文、找到事实。例如，“某个公司的创始人毕业于哪里”。

第二种是结构化数据利用：Agent 能不能查询数据库、知识图谱或表格，并精确组合多个字段。

但现实中的专业决策通常还有第三步：即使事实已经找全，也必须把事实放入一套正式规则中，判断哪些规则适用、哪些规则优先、哪些例外会推翻一般规则。法律、医学、税务、保险、财务合规和海关归类都属于这种问题。

因此，作者提出三层知识复杂度：

| 层级 | 数据类型 | Agent 的主要能力 |
|---|---|---|
| Level 1 | 开放域网页 | 搜索、阅读、总结 |
| Level 2 | 数据库或知识图谱 | 精确查询、结构化知识利用 |
| Level 3 | 层级规则体系 | 处理规则优先级、例外、交叉引用和模糊边界 |

HSCodeComp 专门测试 Level 3。

## 1.2 HS Code 到底是什么

HS Code 是国际贸易商品分类编码。世界范围内通常在前 6 位保持较高一致性，后续位数由各个国家或地区继续细分。论文研究美国税则，因此最终目标是 10 位美国编码。

它不是从几百个互不相关的标签中平面选择一个，而是沿树向下：

```text
商品事实
  ↓
2 位 Chapter：属于哪一章
  ↓
4 位 Heading：章内属于哪个品目
  ↓
6 位 Subheading：品目下属于哪个子目
  ↓
8 位美国细分
  ↓
10 位美国统计细分
```

如果 2 位章选错，后面所有数字自然都会错；即使前 6 位正确，也可能因为国家级细则、材质比例或用途条件而在最后几位失败。

## 1.3 为什么它不是普通商品分类

普通分类模型通常学习 `商品描述 -> 标签` 的统计映射，而 HS 归类需要同时处理：

- **材料**：塑料、橡胶、金属、纺织物等；
- **形态**：原材料、零件、半成品、完整商品；
- **功能**：商品实际做什么；
- **用途**：主要供什么设备或场景使用；
- **主要特征**：组合商品究竟由哪个部分赋予本质；
- **具体列名与兜底条目**：能否使用具体品目，还是只能进入“其他”；
- **部件与整机边界**：商品是独立机器，还是某设备的专用部件；
- **例外与排除**：表面上符合某章，但类注或章注明确要求去另一章。

同一个商品可能在多个维度上指向不同编码，所以正确分类是一个受规则约束的消歧过程。

## 1.4 论文中的输入与输出

论文把任务形式化为：

```text
f(X, R, K) -> Y
```

- `X` 是商品信息：标题、结构化属性、电商类目、图片、价格、币种；
- `R` 是规则：官方层级税则以及专家总结的决策规则；
- `K` 是领域知识：美国海关 CROSS 历史裁定；
- `Y` 是唯一的 10 位编码。

重点在于：答案必须是 HS 分类树上的合法路径，还要满足沿途所有规则约束。

## 1.5 一个直观例子：掌上游戏机

论文展示了一款带 Linux 系统、3.5 英寸屏幕、内置大量游戏的掌上游戏机，商品页面同时包含：

- 冗长营销标题；
- 电商平台三级类目；
- 屏幕、系统、电池等属性；
- 商品图片；
- 价格和币种。

目标是预测 `9504904000`。Agent 不能只看到“电子产品”就进入第 85 章，也不能只看到“显示屏”就按显示设备处理，而必须判断它的完整商品身份是游戏设备，并继续选择美国细分码。

### 本章理解要点

HSCodeComp 测的不是知识记忆，而是下面这条链能否稳定完成：

```text
理解商品 → 找到候选类别 → 检索适用规则 → 处理冲突与例外
→ 沿层级逐步收窄 → 验证编码有效性 → 输出唯一结论
```

## 第二章：数据集为什么称得上“真实、专家级”

对应第二句话：

> 数据集包含 632 个真实商品、32 个类别和 352 个不同编码，由 26 位关税专家通过双人独立标注、专家裁决和抽样复核构建。

## 2.1 数据规模与覆盖

HSCodeComp 包含：

- 632 个真实商品样本；
- 32 个一级电商类别；
- 27 个 2 位章；
- 151 个 4 位品目；
- 268 个 6 位子目；
- 322 个 8 位编码；
- 352 个不同的 10 位编码。

352 个 10 位编码中：

- 256 个编码只出现一次，占 72.73%；
- 58 个出现两次，占 16.48%；
- 其余重复次数更少。

这意味着模型很难依赖训练集式的重复模式，因为大量编码在 Benchmark 里只有一个商品实例。

## 2.2 “真实”体现在哪里

作者依据大型跨境电商平台的销量和类别统计采样，没有先把商品整理成干净的实验室文本，而是有意保留：

- 冗长标题；
- SEO 和营销关键词；
- 不一致的属性；
- 错别字或不规范表达；
- 电商类目与海关分类体系的错位；
- 文本没有写清、但图片中可见的材料和结构。

作者还过滤了“电商类目和 HS 编码都与已有样本相同”的冗余商品，避免简单、畅销、重复商品主导分数。

## 2.3 为什么电商类目不能直接当答案

电商类目是为用户浏览和销售组织设计的，HS 类目是为法律归类和征税设计的。例如：

- 电商会把糖尿病警示手环放在“珠宝配饰”；
- HS 规则可能因为其主要功能是医疗警示，而排除仿首饰分类；
- 电商会把 PU 相机保护壳放在“相机包”；
- HS 分类仍可能更关注具体材料和法律品目边界。

这恰好制造了一个现实难点：平台已有类别是有用线索，但不能成为决定性规则。

## 2.4 专家标注流水线

每个样本大致经过以下步骤：

1. 专家浏览商品页面，收集完整信息；
2. 抽取材料、功能、结构、包装状态等核心事实；
3. 在 CROSS 中搜索类似美国海关裁定；
4. 找到相关案例时，检查商品是否真正相似，并验证裁定编码当前是否有效；
5. 没有可靠先例时，按照官方税则和专家决策规则逐层分类；
6. 在税则系统中验证最终编码确实存在；
7. 两名专家独立完成以上过程；
8. 编码一致则收录，不一致则交给高级专家裁决；
9. 无法形成共识的样本被丢弃。

此外，另一位没有参与初始标注的高级专家重新标注随机 10% 样本，论文报告分歧率为 2%。

## 2.5 六条专家决策规则

论文附录将通用归类逻辑整理为六组规则：

1. **优先依据品目文字和类注、章注。** 如果这些已经足以分类，不应跳到后面的兜底规则。
2. **未完成品与混合材料。** 未完成商品若已具备完整商品主要特征，可按完整商品归类；混合材料要判断是否仍保留某种材料的主要特征。
3. **多个品目竞争时的顺序。** 更具体描述优先，其次判断主要特征，仍无法判断时才考虑数字顺序等后续原则。
4. **最相类似原则。** 前述规则都不能分类时，按最相似商品归类。
5. **包装和容器。** 正常随商品销售的包装通常随商品归类，可重复使用的特殊容器可能单独归类。
6. **子目层级逐级比较。** 进入某一品目后，只能比较同层级子目，再逐层深入，不能跨层跳选。

## 2.6 为什么高质量标签仍不等于绝对无争议

专业规则任务中的“真值”不同于数学答案。即使多位专家一致，也可能存在：

- 商品事实本身不完整；
- 不同证据源互相矛盾；
- 历史裁定与一般规则之间存在边界；
- 税则版本更新；
- 专家对主要特征的判断不同。

因此，这类 Benchmark 最理想的标签形式不应只有一个 10 位数字，还应该包含：

- 被确认的商品事实；
- 候选品目；
- 适用类注和章注；
- 排除其他候选的理由；
- 采用的裁定案例；
- 税则版本日期；
- 专家分歧与裁决记录。

## 2.7 公开 Benchmark 文件的真实结构

官方仓库公开的核心文件是 `data/test_data.jsonl`。它共有 632 行，每一行都是一个独立 JSON 对象，对应一个完整测试任务；`task_id` 从 0 到 631，全部唯一。

每条记录实际包含 17 个字段：

| 字段 | 含义 | 备注 |
|---|---|---|
| `product_name` | 商品标题 | 保留营销词、冗余描述和不规范表达 |
| `product_attributes` | 商品属性 | 在外层 JSON 中被保存为 JSON 字符串，需要再次解析 |
| `price` | 价格 | 数值字段 |
| `currency_code` | 币种 | 如 CNY、USD |
| `cate_lv1_desc` | 一级英文电商类目 | 平台类目，不是 HS 法律分类 |
| `cate_lv1_desc_cn` | 一级中文电商类目 | 与英文类目对应 |
| `cate_lv2_desc` 至 `cate_lv5_desc` | 二至五级英文电商类目 | 部分商品会重复相同末级类目 |
| `cate_lv2_desc_cn` 至 `cate_lv5_desc_cn` | 二至五级中文电商类目 | 用于双语表达 |
| `hs_code` | 专家标签 | 当前文件中为整数，632 条均为 10 位 |
| `task_id` | 样本编号 | 0 至 631 |
| `question` | 完整评测提示 | 重复组织商品信息，并要求输出路径、理由和 `\boxed{}` 编码 |

一个非常重要的实现细节是：`product_attributes` 不是直接嵌套对象，而是一个字符串。例如程序读取一行后，通常还要再执行一次 JSON 解析：

```python
row = json.loads(line)
attributes = json.loads(row["product_attributes"])
```

如果评测代码忘记第二次解析，Agent 看到的属性可能只是带大量转义符的字符串，进一步增加信息处理难度。

## 2.11 从一条记录到实际评测问题

`question` 字段会把记录重组成如下任务：

```text
Given product information, classify the product according to US HSCode standards.

Product Title: ...
Product Attributes: ...
Product Price: ...
Product Category: ...

Requirements:
- output a complete 10-digit HSCode
- use \boxed{}
- provide the complete classification path and rationale
```

提示还允许查询美国海关 CROSS 和 eWTP 税则网站。因此一次评测实际包含两层输出：

- 自动计分主要检查 10 位编码是否精确一致；
- 过程分析再检查搜索、分类路径和理由为何失败。

## 2.12 公开数据的复现注意事项

截至本次核对，官方 GitHub `test_data.jsonl` 中没有图片、图片 URL 或商品页面 URL 字段，Hugging Face 页面也将数据标注为 Tabular/Text；但论文报告了 VLM 和 Agent 使用商品图片的实验。

这可能意味着图片通过未包含在该 JSONL 中的独立内部映射或评测管线提供。对希望完整复现 Table 1 多模态结果的研究者来说，需要进一步寻找作者发布的图片资源或向作者确认映射方式；仅凭当前公开 JSONL 可以直接复现文本任务，但不能自然还原论文所述的图片输入。

### 本章理解要点

论文最重要的资源贡献不是 632 这个数字，而是把昂贵的专家决策过程压缩成一个可重复评测的集合；与此同时，专业标签仍需要开放异议和重裁机制。

## 第三章：当前模型离专家还有多远

对应第三句话：

> 对 23 个 LLM、VLM 和 Agent 的实验表明，最佳 Agent 的准确率仅为 46.8%，远低于人类专家的 95.0%，且错误会随分类层级加深而累积。

## 3.1 被测系统

论文覆盖三类系统。

### A. 基础 LLM/VLM

模型只依赖自身知识和输入内容，包括 GPT-5、Gemini-2.5-Pro、GPT-4o、Claude Sonnet 4、Kimi-K2、DeepSeek-R1、Qwen-Max、o3-mini 等。VLM 版本额外获得商品图片。

### B. 开源 Agent 框架

包括 SmolAgents、AWorld、AgentOrchestra、OWL、WebSailor、Cognitive Kernel。它们可以使用搜索、税则检索和领域知识工具，大多统一使用 GPT-5 作为骨干。

### C. 闭源商业 Agent

包括 Gemini Deep Research、Grok DeepSearch 和 Manus。由于接口等限制，这些系统只在代表性子集上测试，所以结果不能和全量测试作完全严格的一一比较。

## 3.1.1 这组对比究竟在比较什么

理解 Table 1 前，必须先区分三个变量：

```text
骨干模型能力
+ Agent 控制流/规划方式
+ 可调用的外部工具与知识
```

论文尽量控制后两个变量中的一部分：

- 开源 Agent 默认使用 GPT-5 作为骨干；
- 统一提供 Google Search、知识检索和税则检索工具；
- CROSS 通过网页搜索工具访问；
- 专家决策规则和完整商品网页在默认实验中被移除，因为初步实验显示它们降低表现；
- 只有 SmolAgents 的 VLM 配置明确使用了商品图片；
- 基础 LLM/VLM 不使用这些外部工具，主要依赖模型内部知识；
- 闭源商业系统的内部控制流和工具不可见，而且只在一个子集上测试。

因此，Table 1 不是一个完全纯粹的“框架排行榜”。更准确地说：

> 在尽量相同的 GPT-5 骨干和工具条件下，比较不同 Agent 框架如何组织搜索、推理和工具调用；基础模型组则用于测量不使用 Agent 工具时的下限。

另外，AWorld、OWL、WebSailor 等原论文往往同时包含训练方法或自有模型，但 HSCodeComp 报告的是将其作为 Agent 系统/框架并配上统一骨干后的表现，不应把该分数直接理解成其原论文专用模型的能力。

## 3.1.2 方法零：直接使用 LLM/VLM

### 工作方式

```text
商品文本/图片
→ 单次或直接模型推理
→ 生成分类路径和 10 位编码
```

模型没有动态查询税则和 CROSS 的 Agent 工具，主要依赖参数记忆、输入语义和已有常识。

### 代表性结果

| 模型 | 2 位 | 4 位 | 6 位 | 10 位 |
|---|---:|---:|---:|---:|
| GPT-5 VLM | 82.12 | 70.89 | 59.97 | 29.27 |
| GPT-5 LLM | 82.59 | 69.78 | 56.33 | 28.96 |
| Gemini-2.5-Pro VLM | 82.28 | 71.04 | 59.02 | 24.21 |
| GPT-4o VLM | 78.01 | 64.08 | 48.10 | 18.51 |
| Claude Sonnet 4 LLM | 78.80 | 62.97 | 44.94 | 11.87 |
| DeepSeek-R1 | 77.22 | 61.71 | 38.45 | 6.65 |
| o3-mini | 77.22 | 56.17 | 24.53 | 1.27 |

### 应该如何理解

直接模型在 2 位章级仍能达到约 77% 至 82%，说明大模型通常知道商品大概属于机械、塑料、纺织或首饰等哪个大类；但越深入到具体子目，参数记忆中的规则过时、缺失或互相混淆，准确率快速下降。

这一组回答的是：**只靠一个强模型“凭知识回答”，能走多深？**

## 3.1.3 SmolAgents：轻量、直接执行的工具 Agent

### 原始框架特点

SmolAgents 是 Hugging Face 的轻量 Agent 库，强调少量抽象、模型无关和工具无关。其代表性 `CodeAgent` 会把行动写成代码，直接调用搜索或其他工具，而不是维护非常复杂的多 Agent 规划层。

官方项目：https://github.com/huggingface/smolagents

### 在 HSCodeComp 中的工作方式

可以将它近似理解为：

```text
读取商品
→ 根据当前缺口形成搜索/税则查询
→ 直接调用工具
→ 读取返回结果
→ 继续查询或提交编码
```

它既有文本版，也有加入商品图片的视觉版。

### 结果

| 配置 | 2 位 | 4 位 | 6 位 | 8 位 | 10 位 |
|---|---:|---:|---:|---:|---:|
| SmolAgents LLM | 82.28 | 70.89 | 59.81 | 49.05 | 42.72 |
| SmolAgents VLM | 82.06 | 72.06 | 62.38 | 52.38 | **46.83** |

### 为什么在本任务中最好

论文将 SmolAgents 称为更接近 direct execution 的框架。它的优势不是规划更宏大，而是：

- 能使用当前税则和 CROSS，减少模型记忆过时；
- 工具调用路径相对短，较少在内部推理中发明额外约束；
- 视觉版能补充部分材料与表面信息；
- 不容易像重推理框架那样在多轮自我解释中漂移。

但 46.83% 仍说明直接执行并没有解决规则优先级和细粒度边界问题。

## 3.1.4 AWorld：面向大规模 Agent—环境交互的系统

### 原始方法特点

AWorld 原论文关注大规模 Agent 与环境交互、经验生成和训练流水线。它通过分布式执行加速轨迹采集，使 Agent 强化学习和经验学习更容易扩展。它更像 Agent 运行与训练基础设施，而不只是一个固定提示词模板。

原论文：https://arxiv.org/abs/2508.20404

### 在 HSCodeComp 中的含义

HSCodeComp 使用 AWorld 框架并配置 GPT-5 和统一工具，主要测试其任务编排、交互循环和工具使用能力，而不是测试 AWorld 原论文中训练得到的 Qwen3-32B Agent。

### 结果

```text
2 位 82.28
4 位 70.41
6 位 59.18
8 位 48.58
10 位 41.30
```

它与 SmolAgents 文本版接近，但最后 10 位低 1.42 个百分点。说明良好的 Agent—环境交互基础设施能带来稳定工具使用，却不会自动提供海关规则所需的确定性决策逻辑。

## 3.1.5 AgentOrchestra：中央规划器协调专业子 Agent

### 原始方法特点

AgentOrchestra 是层级多 Agent 框架：中央 Planner 分解任务并协调专业子 Agent。其后续版本提出 Tool-Environment-Agent（TEA）协议，将工具、环境和 Agent 建模成带版本和生命周期的资源，以加强上下文管理、可追踪性和能力扩展。

原论文：https://arxiv.org/abs/2506.12508

### 可想象的任务分工

```text
中央规划器
├─ 商品事实分析 Agent
├─ 税则搜索 Agent
├─ CROSS 案例检索 Agent
└─ 汇总/决策 Agent
```

### 结果

```text
2 位 82.12
4 位 70.73
6 位 60.44
8 位 47.78
10 位 41.30
```

它的 6 位准确率略高于 AWorld 和 SmolAgents 文本版，但 8 到 10 位下降明显。一个合理解释是，多 Agent 分工有助于覆盖不同信息源，却仍需要一个严格的最终规则裁决器；否则多个 Agent 提供的候选和理由只会增加汇总阶段的冲突。该解释是根据架构与结果作出的推断，不是论文直接验证的因果结论。

## 3.1.6 OWL：Planner—Coordinator—Workers 的层级 Workforce

### 原始方法特点

OWL 的运行框架称为 Workforce，由三层组成：

1. 领域无关 Planner：拆解任务；
2. Coordinator：管理子任务和工作流；
3. 专业 Workers：使用特定领域工具完成子任务。

OWL 本身还包含用现实反馈强化学习优化通用 Planner 的训练方法，目标是让多 Agent 系统更容易跨领域迁移。

原论文：https://arxiv.org/abs/2505.23885

### 结果

```text
2 位 72.63
4 位 61.87
6 位 51.58
8 位 41.77
10 位 37.34
```

它的特点是 2 位准确率就明显低于前三个框架，表明部分任务在高层规划或初始任务路由阶段已经选错方向。一旦 Planner 将商品错误分配到某类解释框架，后续 Worker 即使检索充分也很难返回正确路径。

注意：这里评测的是 HSCodeComp 的 GPT-5 框架配置，不能据此否定 OWL 原论文训练出来的专用 Planner 在其他任务上的效果。

## 3.1.7 WebSailor：为高不确定网页搜索训练的重推理 Agent

### 原始方法特点

WebSailor 的目标是让 Web Agent 在巨大信息空间中系统性降低不确定性。其方法包括高不确定任务构造、信息混淆、RFT 冷启动，以及名为 DUPO 的 Agent 强化学习算法。它特别强调复杂搜索中的深层推理。

原论文：https://arxiv.org/abs/2507.02592

### 结果

```text
2 位 81.64
4 位 70.56
6 位 57.27
8 位 43.98
10 位 35.44
```

WebSailor 在 2、4 位仍不错，但越深入下降越明显。论文用它说明“适合开放域搜索的长推理，不一定适合严格规则应用”：

| 推理设置 | 10 位准确率 |
|---|---:|
| Overthink | 35.44 |
| Medium-Think | 37.34 |
| No-Think，直接调用工具 | 40.82 |

WebSailor 原本擅长在开放网页中逐渐缩小不确定性；HSCodeComp 却要求在已给定的法律规则中保持确定性约束。更多开放式推理可能把模糊商品细节扩写成不存在的条件，造成 reasoning drift。

## 3.1.8 Cognitive Kernel-Pro：多模块研究与测试时反思

### 原始方法特点

Cognitive Kernel-Pro 是开放的多模块 Deep Research Agent 框架，覆盖 Web、文件、代码和一般推理，并研究测试时反思和投票。其目标之一是降低对付费 API 和封闭工具的依赖，同时形成可训练的 Agent Foundation Model 数据与轨迹。

原论文：https://arxiv.org/abs/2508.00414

### 结果

```text
2 位 80.06
4 位 69.15
6 位 54.59
8 位 40.03
10 位 26.42
```

它在 2 位大类仍接近其他系统，但 10 位仅 26.42%，甚至低于 SmolAgents 很多。一个值得注意的结构错配是：Cognitive Kernel-Pro 原本强调多模块研究、反思和投票，而 HSCodeComp 的实验恰好发现开放式反思与多数投票无效或有害。这里同样只能说明该配置与本任务不匹配，不能把原因完全归结于某个单独模块。

## 3.1.9 闭源 Deep Research：Gemini、Grok 与 Manus

### 共同特点

三者都是黑盒商业系统，通常会自主规划、浏览网页、读取多个来源并生成综合报告。论文无法统一它们的内部模型、搜索策略、反思次数或工具实现。

### 子集结果

| 系统 | 10 位准确率 |
|---|---:|
| SmolAgents + GPT-5 | 42.86 |
| AWorld + GPT-5 | 42.86 |
| Gemini Deep Research | 40.81 |
| Manus | 30.61 |
| Grok DeepSearch | 26.53 |

这些数字来自代表性子集，不是完整 632 题，因此不能与 Table 1 的全量 46.83% 直接相减比较。

论文的案例显示，商业系统常见问题包括：

- 在证据不足时提前锁定商品类别；
- 收集到正确细节，却在汇总阶段误解其意义；
- 生成听起来完整但没有事实依据的功能描述；
- 通过自我修正把最初正确答案改错。

开源 Agent 在该子集上略好，并不意味着它们整体比商业系统更强，而是说明通用 Deep Research 的长报告和广泛浏览优势，无法自然转化成严格的规则遵循。

## 3.1.10 六种开源 Agent 的横向比较

| 方法 | 原始设计重心 | 本文中的控制风格 | 10 位准确率 | 在本任务中的主要现象 |
|---|---|---|---:|---|
| SmolAgents VLM | 轻量代码 Agent、直接工具调用 | 短路径直接执行 | **46.83** | 最少推理漂移，图片有净收益 |
| SmolAgents LLM | 同上，无图片 | 短路径直接执行 | 42.72 | 文本 Agent 中最佳 |
| AWorld | Agent—环境交互与经验生成基础设施 | 交互式工具编排 | 41.30 | 稳定，但没有专门规则裁决器 |
| AgentOrchestra | 中央 Planner + 专业子 Agent | 层级多 Agent | 41.30 | 信息覆盖好，最终细分仍困难 |
| OWL | Planner + Coordinator + Workers | 层级任务分解 | 37.34 | 高层路由错误较多 |
| WebSailor | 高不确定 Web 搜索与深推理 | 重推理搜索 | 35.44 | 越想越偏；No-Think 升至 40.82 |
| Cognitive Kernel | 多模块研究、反思、投票 | 研究型多步流程 | 26.42 | 反思/投票风格与规则任务错配 |

## 3.1.11 从方法对比中能得到什么

第一，**工具增强是必要的**。所有强 Agent 都显著超过同骨干直接回答的 10 位结果。

第二，**多 Agent 或更复杂规划不是充分条件**。AgentOrchestra、OWL 等具备层级编排，但仍没有超过更简单的 SmolAgents。

第三，**开放域搜索能力和规则应用能力不是一回事**。WebSailor 擅长通过长推理降低网页搜索不确定性，却在严格规则树中出现推理漂移。

第四，**最终缺少的不是更多信息，而是规则裁决器**。这些方法都能搜索税则和案例，但没有框架真正把规则优先级、排除条款和逐层约束编译成不可跳过的控制逻辑。

所以这组对比最终支持论文的核心诊断：

> 当前 Agent 框架的差别主要影响“怎样寻找和组织信息”，但 HSCodeComp 的真正瓶颈是“怎样把信息约束在一个严格、逐层、可验证的规则执行过程中”。

## 3.2 为什么采用多层 Exact Match

论文分别报告 2、4、6、8、10 位准确率。

- 2 位正确：大方向正确；
- 4 位正确：已经找到相对具体的商品身份；
- 6 位正确：国际统一层级下的细分类大体正确；
- 8/10 位正确：能够处理美国特定税则细节。

10 位 Exact Match 最接近真实申报需要，但它会把“只错最后两位”和“第一章就选错”都记作同一个错误。因此论文又计算条件准确率，用来分析错误具体发生在哪一层。

## 3.3 最佳结果及其含义

最佳系统 SmolAgents + GPT-5 + Vision 的结果为：

| 层级 | 准确率 |
|---|---:|
| 2 位 | 82.06% |
| 4 位 | 72.06% |
| 6 位 | 62.38% |
| 8 位 | 52.38% |
| 10 位 | 46.83% |

人类专家的 10 位准确率为 95.0%。

这里最值得注意的不是单个 46.83，而是准确率随深度稳定下降：Agent 经常能判断商品大类，却无法持续处理后续的材料比例、用途、排除条款和国家细分规则。

## 3.4 Agent 为什么比单模型强

GPT-5 单模型的结果：

- VLM：10 位 29.27%；
- LLM：10 位 28.96%。

加入 Agent 框架和工具后，最佳结果达到 46.83%。论文进一步统计，SmolAgents 相比直接使用 GPT-5，主要减少了：

- 事实性错误；
- 使用已经过时的编码；
- 没有规则依据的推理幻觉。

这说明 Agent 的主要价值不是“替模型增加智力”，而是允许模型接触当前规则、历史裁定和验证工具。

## 3.5 为什么错误会逐层积累

假设一个系统在每一层都有 85% 的条件正确率，连续完成四五层后，总体成功率也会显著下降。实际任务中各层还不是独立的：

- 上层选错会完全关闭正确分支；
- 上层选对但理由错误，可能在下层再次偏离；
- 更深层的规则通常更细、更依赖上下文；
- 8 到 10 位又加入美国特定分类要求。

所以层级任务需要的不只是每一步“平均还不错”，而是整条路径持续可靠。

## 3.6 类别表现说明什么

论文发现多数商品类别的平均 10 位准确率低于 25%，只有 Hair Extensions & Wigs 相对较高，约 47%。即使 Jewelry & Accessories、Home & Garden、Tools 等样本较多的类别，平均表现也仍低于 18%。

难题还集中在 Novelty & Special Use、Men's Clothing 等长尾类别。这表明困难并不只是训练样本少，也来自类别边界和非标准描述。

### 本章理解要点

Agent 已经能够通过工具减少知识过时和明显幻觉，但在“沿规则树持续保持正确”这件事上仍远未达到专业可用程度。

## 第四章：哪些增强方法有效，哪些导致推理漂移

对应第四句话：

> 消融实验发现官方税则至关重要，图片和历史裁定可提供辅助，但简单加入专家规则、增加推理步骤、多数投票或自我反思反而可能造成“推理漂移”。

## 4.1 官方税则是任务主干

SmolAgents + GPT-5 VLM 的相关结果：

| 配置 | 10 位准确率 |
|---|---:|
| 不使用专家决策规则 | 46.83% |
| 加入专家决策规则 | 43.83% |
| 移除官方税则 | 36.73% |
| 移除 CROSS | 46.53% |

移除官方税则下降 10.10 个百分点，说明 HSCodeComp 本质上是规则应用任务，而不是相似商品检索任务。

## 4.2 为什么历史案例帮助有限

CROSS 裁定可以告诉 Agent：美国海关过去如何处理某类具体商品。但案例检索有几个问题：

- 搜到的商品可能只在表面上相似；
- 裁定可能基于不同材料或用途；
- 某些裁定已经被撤销；
- 旧裁定中的编码可能失效；
- 相似案例不能推翻当前税则的明确文字。

因此案例更适合在候选品目已经形成后解决局部歧义，而不适合作为一开始就照抄答案的最近邻分类器。

## 4.3 为什么加入“专家规则”反而下降

直觉上，把六条专家决策规则放进提示词应该有帮助，但实际从 46.83% 降到 43.83%。可能原因包括：

1. 规则是抽象原则，商品事实却高度具体且含噪；
2. 模型无法稳定判断某条规则何时触发；
3. 多条规则、税则原文和历史案例同时进入上下文，产生信息过载；
4. 模型把规则当作可以自由解释的文本，而不是必须满足的程序约束；
5. “主要特征”等概念本身仍需要专业判断。

这告诉我们：**规则存在于上下文中，不等于规则被执行。**

## 4.4 图片：有帮助，也会制造干扰

| 模型 | 无图片 | 有图片 | 提升 |
|---|---:|---:|---:|
| GPT-5 Agent | 42.72% | 46.83% | +4.11 |
| Claude Sonnet 4 Agent | 33.70% | 34.65% | +0.95 |
| GPT-4o Agent | 22.03% | 22.31% | +0.28 |

进一步分析显示，图片让 6.42% 原本错误的文本样本变正确，却也让 6.17% 原本正确的样本变错。

原因是图片确实可能补充材质、表面和组装状态，但通用视觉编码器也可能误认专业细节，并让不可靠的视觉判断覆盖明确文本。

更合理的方法是把视觉输出当作一条带置信度的证据，而不是让 VLM 隐式融合后直接决定答案。

## 4.5 Test-Time Scaling 为什么失败

作者测试了两种常见策略。

### 多数投票

对同一问题运行 `K = 1, 2, 4, 8, 16` 次，再用多数结果作为答案。结果几乎没有改善。

因为如果模型存在系统性偏差，多次采样只会重复同一个错误；而 10 位编码候选极多，错误答案还可能分散，使简单多数票不稳定。

### 自我反思

让 SmolAgents 主动检查并修订自己的推理，准确率从 42.72% 略降到 42.57%。

开放式反思没有引入新的可靠事实或规则，只给模型更多机会怀疑正确结论、发明额外条件，最终可能把正确答案改错。

## 4.6 推理漂移是什么

论文用 reasoning drift 描述这样的过程：

```text
初始事实大致正确
→ 在长推理中加入未经证实的解释
→ 把模糊细节当成确定条件
→ 搜索逐渐围绕错误假设展开
→ 新搜索结果被用来确认已有偏见
→ 最终偏离正确规则路径
```

WebSailor 的消融非常典型：

- 过度推理：35.44%；
- 中等推理：37.34%；
- No-Think，直接调用工具：40.82%。

对于知识和约束主要位于外部规则中的任务，少做无依据的内部推演、更多使用精确工具，反而更好。

## 第五章：通过论文案例真正理解失效模式

这一章集中展开论文中的代表性商品。

## 5.1 案例一：PVC 手套——标题与技术属性冲突

### 商品信息

标题包含“Pink Nitrile Gloves”，容易让人理解为丁腈橡胶手套；但结构化属性明确写着：

```text
Material: PVC
```

### 错误路径

Agent 优先相信标题中的 Nitrile，搜索 `4015.19`，将商品归入第 40 章橡胶制品，还输出了一个过时编码。

### 正确路径

1. 标题是营销性自然语言，可靠度较低；
2. `Material: PVC` 是明确技术属性，可靠度较高；
3. PVC 应按塑料处理；
4. 商品是塑料制一次性手套；
5. 最终进入 `3926201020`。

### 这个案例教会我们什么

Agent 在推理前必须建立证据优先级：

```text
可验证技术规格 > 清晰结构化属性 > 图片推断 > 商品标题 > 营销文案
```

这不是绝对顺序，但必须显式处理冲突，而不能让更醒目的词自动获胜。

## 5.2 案例二：糖尿病警示手环——功能压过外形

### 商品信息

商品外形是硅胶手环，上面写有“TYPE 2 DIABETIC”，电商平台把它放在珠宝与配饰类别。

### 错误路径

Agent 看到 bracelet、fashion、jewelry 等词，将其归入 `7117` 仿首饰。

### 正确路径

1. 外形确实是手环；
2. 但“TYPE 2 DIABETIC”表明它承担医疗警示功能；
3. 核心用途不是装饰，而是向他人传递健康信息；
4. 功能性使其不应按普通仿首饰处理；
5. 再结合硅胶/塑料材料，进入第 39 章；
6. 最终标签为 `3926909989`。

### 这个案例教会我们什么

分类不是找最像的名词，而是比较：

```text
商品叫什么？
商品看起来像什么？
商品由什么制成？
商品主要做什么？
哪一个特征赋予它本质？
```

## 5.3 案例三：PU 相机半包——第一次搜索锁死路径

### 商品信息

商品是 Olympus PEN-F 相机使用的 PU Leather 半包。电商类目和标题都强调 camera case。

### 错误路径

Agent 第一条查询就是：

```text
HTSUS camera case 4202.92 ...
```

这等于在检索前已经决定 Heading 4202，后面只寻找支持 4202 的证据，没有重新检查 PU 材料是否改变分类。

### 正确标签

论文给出的正确标签是 `3926909989`，按塑料制其他物品处理。

### 这个案例教会我们什么

搜索查询不是中性的，它会决定 Agent 接下来看到什么。更好的第一轮查询应并行探索：

```text
候选 A：按相机盒/包的具体品目
候选 B：按 PU 塑料制品
候选 C：是否存在章注或排除条款
```

然后分别寻找支持与反对证据，而不是一开始就把一个候选写进搜索词。

## 5.4 案例四：PVC 装饰贴纸——营销噪声改变“主要特征”判断

### 商品信息

标题包含 INS、Aesthetic、Korean Stationery、Decoration、Scrapbooking 等大量风格和用途词，结构化属性则写明 PVC、自粘贴纸。

### 错误路径

Agent 把印刷图案视为商品主要特征，归入第 49 章印刷品 `4911914040`。

### 正确路径

商品的法律身份更受自粘 PVC 材料和形态约束，因此正确标签为 `3919102055`。

### 这个案例教会我们什么

输入解析时应先给词语分角色：

- 技术事实：PVC、自粘、卷状或片状；
- 商品功能：装饰、粘贴；
- 营销叙事：INS、Aesthetic、Korean；
- 交易信息：数量、价格、包装。

营销词可以帮助理解使用场景，却不应与技术材料同权。

## 5.5 案例五：叉车安全笼——反思怎样把问题想坏

SmolAgent 依据 Section XVI Note 2，判断安全笼专门或主要供叉车使用，因此作为叉车部件归入 `8431200000`。

其他系统出现三种错误：

- 过度纠结“部件还是附件”，忽略更直接的专用性规则；
- 凭空把被动金属笼解释为有独立机械功能的机器；
- 把为了地面搬运设置的小轮误解成车辆底盘。

这个案例说明，反思应该是检查清单式的局部验证：

1. 商品是否真的具备刚才声称的特征？
2. 是否有逐字规则支持结论？
3. 是否跳过了更高优先级的规则？
4. 是否把次要特征放大成主要功能？

而不是简单说“请再深入思考”。

## 5.6 论文归纳的六类失败

| 失败类型 | 表现 | 典型后果 |
|---|---|---|
| 过早决策 | 证据不足就锁定高层类别 | 后续检索只确认错误方向 |
| 信息误处理 | 忽略或误读关键字段 | 营销词压过材料属性 |
| 不必要自我修正 | 初始答案正确，反思后改错 | 推理越长越差 |
| 推理幻觉 | 编造商品功能或法律条件 | 理由看似完整但无事实基础 |
| 错误规则应用 | 规则优先级或适用范围错误 | 形式压过主要功能 |
| 缺乏领域知识 | 不懂材料、部件、用途等专业边界 | 选错章或使用过时代码 |

## 5.7 失败案例对照表：叉车安全笼

公开数据 Task 178：

```text
商品：36×36 英寸可折叠叉车安全笼/工作平台
承重：1200 lbs
特征：带 4 个万向轮
电商类目：Furniture -> Outdoor Furniture -> Garden Furniture Sets
专家标签：8431200000
```

这里连电商类目本身都是强噪声：商品被放进“花园家具套装”，但真实身份是专门配合叉车使用的工作平台。

论文比较了四种系统：

| 系统 | 最终编码 | 核心判断 | 失败原因 |
|---|---:|---|---|
| SmolAgent | `8431200000` | 专门或主要供叉车使用，应按叉车部件处理 | 正确 |
| Gemini Deep Research | `7326908688` | 认为它不是不可缺少的部件，只是钢制附件 | 不必要自我修正；忽略更直接的专用部件规则 |
| Grok DeepSearch | `8428900290` | 把安全笼解释成有独立机械功能的装置 | 推理幻觉；虚构商品并不具备的主动机械能力 |
| Manus | `8427900020` | 把万向轮解释成移动式车辆底盘 | 信息误处理；把方便搬运的小轮夸大为完整车辆结构 |

这个案例非常典型，因为所有错误理由都“听起来专业”，但分别违反了不同基本要求：

- Gemini 没有优先应用直接控制该商品的规则；
- Grok 在事实层虚构功能；
- Manus 正确看见了轮子，却错误理解轮子的作用。

## 5.8 失败案例对照表：工业缓冲器被误认成动力装置

公开数据 Task 475：

```text
商品：HR60/80 Pneumatic Cylinder Shock Absorber
属性中的产品名：Hydraulic buffer
材料：iron
用途：自动化机械
功能：降低振动和噪声、延长机器寿命
专家标签：8487900080
```

商品标题同时出现 `Pneumatic Cylinder` 和 `Shock Absorber`，但属性明确说明它是被动吸收能量的缓冲器，而不是产生动力的执行器。

| 系统 | 最终编码 | 系统理解 | 失败类型 |
|---|---:|---|---|
| SmolAgent | `8487900080` | 排除更具体品目后，作为未在其他地方列名的通用机械部件 | 正确 |
| Gemini Deep Research | `8412210075` | 把线性液压装置理解为产生运动的液压马达 | 信息误处理：把耗散能量误作产生动力 |
| Grok DeepSearch | `8302496085` | 认为它只是普通金属附件 | 过早决策：没有充分比较第 83 章与第 84 章 |
| Manus | `8412310080` | 被标题中的 Pneumatic Cylinder 诱导，认作气动执行器 | 信息误处理：关键词压过明确功能属性 |

这个例子揭示了 HS 分类里非常常见的“功能极性”问题：

```text
主动产生力或运动 ≠ 被动吸收冲击或耗散能量
```

两种设备在外形和技术词汇上可能很像，但法律分类取决于真实工作机理。

3. 论文图表或数据版本不一致。

只有第三方能看到逐题运行结果和专家裁决链，才能可靠地区分三者。

## 第六章：论文真正给出的 Agent 设计原则

对应第五句话：

> 专业 Agent 的关键不是想得更久，而是采用结构化事实抽取、可执行规则、逐层验证和可追溯证据链来约束推理。

## 6.1 先构造商品事实表

不要直接把原始商品页面交给一个长 CoT。第一步应产生结构化事实：

| 事实 | 值 | 来源 | 可靠度 | 冲突 |
|---|---|---|---|---|
| 材料 | PVC | 结构化属性 | 高 | 与标题 Nitrile 冲突 |
| 商品形态 | 一次性手套 | 标题、图片 | 高 | 无 |
| 用途 | 清洁 | 属性 | 中 | 无 |
| 电商类目 | 家居清洁 | 平台 | 低到中 | 不是法律类目 |

推理系统必须先解决或保留冲突，不能在隐藏状态里自动混合。

## 6.2 候选生成和候选裁决分离

阶段一生成多个高召回候选：材料候选、功能候选、具体品目候选、部件候选。

阶段二对每个候选检查：

- 品目文字是否直接覆盖商品；
- 类注或章注是否排除；
- 是否有更具体品目；
- 主要特征是否支持；
- 历史裁定是否真的同构；
- 当前编码是否有效。

这样可以减少第一个想法控制整个搜索过程。

## 6.3 规则要成为可执行对象

理想规则表示不只是自然语言段落，而应该带有：

- 作用范围；
- 前置条件；
- 结论；
- 优先级；
- 排除条件；
- 交叉引用；
- 时间版本；
- 需要的商品字段。

例如可以把“专门或主要用于某机器的部件”转成待验证条件，而不是让模型凭语感引用。

## 6.4 沿层级提交与验证

每深入一层都生成一个小型审计记录：

```text
当前层：4 位 Heading
选择：3926
父级：39，已验证
同层候选：3919、3926
排除 3919 的理由：商品并非该形态/条件
支持 3926 的规则：……
待解决字段：是否自粘、是否成卷
```

只有当前层闭环后，才允许进入下一层。

## 6.5 反思需要触发器和停止条件

适合触发反思的情况：

- 标题与结构化属性冲突；
- 两个候选都得到规则支持；
- 搜到的历史裁定已撤销或版本过旧；
- 编码在当前税则中不存在；
- 图片与文本冲突；
- 最终路径中某层没有证据。

停止条件可以是：

- 每层都有有效规则依据；
- 关键冲突已经解决或明确标为未知；
- 编码通过当前版本验证；
- 没有未检查的高优先级排除条款。

## 6.6 最终输出应可审计

真实业务输出不应只有：

```text
3926201020
```

而应至少包括：

```text
最终编码：3926201020
核心事实：材料 PVC；商品为一次性手套
事实来源：结构化属性、图片
分类路径：39 → 3926 → 392620 → 39262010 → 3926201020
关键规则：……
排除候选：4015，原因是商品材料不是橡胶
税则版本：……
需要人工复核：标题写 Nitrile，与材料属性冲突
```

# 第七章：对论文方法和 Benchmark 的改进建议

## 7.1 Agent 方法改进

### 固定控制流，而不是完全自主规划

使用固定阶段：

```text
事实抽取
→ 冲突检测
→ 候选章/品目生成
→ 规则检索
→ 排除与比较
→ 逐层子目选择
→ 编码有效性检查
→ 证据链输出
```

LLM 只在每个窄阶段解决语义问题，控制器负责流程和不变量。

### 规则编译与规则图

把税则转为机器可遍历的规则图，对交叉引用、例外和优先级做显式建模。LLM 可以解释模糊概念，但不能任意跳过约束。

### 双向验证

一条路径从商品事实自顶向下寻找编码；另一条路径从候选编码反向检查商品是否满足全部条件。两者一致后再提交答案。

### 反证优先

要求 Agent 对每个候选先搜索排除条款和反例，而不是只寻找支持材料，降低确认偏误。

### 时间感知检索

所有税则、编码和 CROSS 裁定都必须带时间戳、有效期和撤销状态。使用旧案例时要把法律逻辑与旧编码分离。

### 人机协作和选择性预测

当证据冲突或置信度不足时，系统应允许：

- 输出 Top-k 候选；
- 指出决定候选差异的缺失字段；
- 请求补充材质、用途或组装状态；
- 将高风险样本交给专家。

## 7.2 Benchmark 改进

### 更丰富的指标

除了 10 位 Exact Match，还可以报告：

- 路径距离；
- 首次出错层级；
- 每层条件准确率；
- 规则引用正确率；
- 商品事实忠实度；
- 幻觉属性数量；
- 置信度校准；
- 覆盖率—准确率曲线；
- 人工复核工作量。

### 发布完整裁决链

为每题提供候选、适用规则、排除规则和专家争议点，使研究者能判断模型错误还是标签有争议。

### 多法域和多时间版本

前 6 位全球相对统一，8 到 10 位具有国家特性。未来可加入中国、欧盟等税则，并设置年度版本迁移任务。

### 对抗式冲突集

单独构造：

- 标题与属性冲突；
- 文本与图片冲突；
- 电商类目与法律类目冲突；
- 新旧编码冲突；
- 多功能组合商品；
- 部件与附件边界。

这能把总体准确率分解成更具体的能力诊断。

## 第八章：阅读论文时必须注意的疑点

## 8.1 Figure 13 内部描述相反

图中表格写：

- Ground truth：`8526910040`；
- 无专家规则版本正确预测 `8526910040`；
- 有专家规则版本错误预测 `8543709860`。

但紧接着的 Failure Analysis 却把两种路径的描述对调了；而官方公开 `test_data.jsonl` 中 Task 31 的标签实际是 `8543709860`。因此 Figure 13 至少有一处 ground truth、系统结果或正确性标记被写反，不能根据该图判断究竟是哪种机制如何误用规则。

## 8.2 数据时间锚点不一致

Limitation 写的是 2025-06-30，Appendix G 写的是 2024-06-30。复现实验前应以实际数据文件、税则快照或作者勘误为准。

## 8.3 CROSS 消融数字不一致

Table 11 显示移除 CROSS 后从 46.83% 变为 46.53%，下降 0.30 个百分点；附录正文却写下降 4.61%。总体“CROSS 是辅助信息”的结论仍成立，但具体幅度需要核验。

## 8.4 闭源系统比较不是完全同条件

商业系统因为 API 可用性只测试代表性子集，不能简单把它们与全量 632 样本上的开源 Agent 数字作严格排名。

## 8.5 后续独立工作对标签提出异议

2026 年 5 月的一篇独立预印本提出确定性六阶段工作流，并在 HSCodeComp 的 6 位层级报告 64.2% Top-1。该工作对 226 个分歧样本进行人工审计，初步认为部分标签可能偏离 HS 通则。

这项后续结果不能单独推翻原论文，因为不同法域、规则版本和审计标准都可能影响判断；但它强化了一个改进方向：专业 Benchmark 应发布完整裁决依据，并允许公开重裁。

## 第九章：如何把论文迁移到法律、医疗和税务 Agent

HSCodeComp 的一般化结构是：

```text
现实世界含噪事实
  +
层级专业规则
  +
历史案例或领域知识
  ↓
唯一、合规、可审计的决定
```

## 法律

- 事实抽取对应案件事实；
- 税则对应法律条文及司法解释；
- CROSS 对应判例；
- HS 路径对应法律要件逐层满足；
- 主要风险同样是过早定性、忽略例外和引用过时规则。

## 医疗

- 商品属性对应患者症状、检查和病史；
- 层级规则对应临床指南和诊断标准；
- 历史裁定对应既往病例或研究证据；
- 关键是适应证、禁忌证、排除诊断和证据等级。

## 税务与财务合规

- 商品事实对应交易事实；
- 税则对应税法、会计准则和监管条款；
- 历史案例对应监管问答和处罚案例；
- 需要严格处理适用范围、生效日期和例外。

共同的系统架构应是：

```text
规则引擎负责约束
LLM 负责语义判断
检索系统负责证据
验证器负责防止漂移
人类专家处理高风险分歧
```

## 第十章：最终评价

## 论文最强的地方

1. 把“规则应用”明确提出为区别于网页搜索和数据库查询的 Agent 能力层级；
2. 选取了一个真实、高风险、具有明确层级结构的专业任务；
3. 使用 26 位专家进行高成本标注和验证；
4. 不只报告排行榜，还系统研究推理深度、图片、税则、历史案例和专家规则；
5. 揭示“更多推理可能更差”这一对专业 Agent 很重要的现象。

## 论文仍不足的地方

1. 数据量有限，难以覆盖全球贸易的极端长尾；
2. 只覆盖美国税则，不能直接代表跨法域泛化；
3. Exact Match 无法充分解释每一类过程错误；
4. 部分案例、日期和消融数字存在内部不一致；
5. 专家裁决链公开程度仍不足，社区难以独立复核争议标签；
6. 模型和 Agent 默认参数不完全一致，复现时需要更严格控制。

## 最值得记住的一句话

> HSCodeComp 证明，专业 Agent 的瓶颈不是“不会生成足够长的推理”，而是“不能持续用可靠事实和正式规则约束每一步决策”。

如果把论文浓缩成一个工程建议，就是：

```text
不要让通用 Agent 自由思考后直接给答案；
要让它在一个可执行、可验证、可停止、可审计的规则流程中工作。
```

- **Heading**：4 位品目。
- **Subheading**：6 位子目。
- **CROSS**：美国海关裁定在线搜索系统，包含历史商品归类裁定。
- **Exact Match Accuracy**：预测编码必须与标签完全相同才算正确。
- **Conditional Accuracy**：在上一级编码已经正确的条件下，下一级继续正确的比例。
- **Essential Character**：主要特征；决定组合或多属性商品本质的材料、功能或用途。
- **Premature Decision**：过早锁定分类路径。
- **Reasoning Drift**：推理过程中逐步引入无依据假设，偏离正确事实和规则路径。
- **Test-Time Scaling**：测试时通过多次采样、投票、更长推理或反思增加计算量。
