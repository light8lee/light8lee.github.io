<h2>论文身份：ERL</h2>

<h2>核心问题：Agent 到了新环境就归零</h2>

今天的通用 Agent 有个通病：到了新环境，不管之前做过多少类似的事，它都从零开始。这不是因为模型不够大，而是因为 Agent 体系缺少一个持续积累、检索和复用经验的机制。微调能解决一部分问题，但对闭源模型不可行，而且不支持持续学习。
</div>
</section>

<figure>
<img src="{{ '/assets/posts/video-notes/2026-06-25-erl/images/04.png' | relative_url }}" alt="现有方法的瓶颈" loading="lazy">
</figure>
<div markdown="1">
<p class="visual-note-index">004 / Video Notes</p>
<h2>现有方法的瓶颈</h2>

之前也有人想过让 Agent 长记性。ExpeL 的做法是反复重试同一个任务，直到成功，然后把成功和失败的轨迹对比，提取出可复用的洞察。问题是，这些洞察会被无脑塞进每一个测试提示里，经验一多就爆炸。AutoGuide 的做法更细，它在离线训练任务上建立上下文感知的指南，测试时每一步都做上下文识别和指南检索。但开销很大，而且如果当前状态匹配不上任何存储的上下文，Agent 就得不到任何指导。更关键的是，这两个方法都需要对同一个任务做多次 rollout，构造对比轨迹对，这在真实部署里往往做不到，因为很多任务根本不能重试。
</div>
</section>

<div markdown="1">
<p class="visual-note-index">005 / Video Notes</p>
<h2>ERL 总览：两步闭环</h2>

ERL 的设计目标很明确：只用单次尝试的轨迹，就能生成可复用的经验，并且在测试时精准检索、注入指导。它分成两步。
</div>
</section>

<section class="visual-note" markdown="1">
<figure>
<img src="{{ '/assets/posts/video-notes/2026-06-25-erl/images/06.png' | relative_url }}" alt="启发式生成：单次轨迹即可" loading="lazy">
</figure>
<div markdown="1">
<p class="visual-note-index">006 / Video Notes</p>
<h2>启发式生成：单次轨迹即可</h2>

第一步叫启发式生成。Agent 完成一个任务后，不管成功还是失败，都会做一次反思。反思产出的是一个结构化启发式，包含两部分：一是分析，指出这次成败的根本原因；二是学习指南，用明确的触发条件和推荐动作写成一句可操作的话，比如发邮件给日历参与者之前，先用 Contacts 工具把名字解析成邮箱地址。这些启发式被存进一个持久化池子里。
</div>
</section>

<section class="visual-note" markdown="1">
<figure>
<img src="{{ '/assets/posts/video-notes/2026-06-25-erl/images/07.png' | relative_url }}" alt="检索增强执行：只注入最相关的 top-k" loading="lazy">
</figure>
<div markdown="1">
<p class="visual-note-index">007 / Video Notes</p>
<h2>检索增强执行：只注入最相关的 top-k</h2>

第二步叫检索增强执行。面对新任务时，Agent 不会让经验池全部涌进上下文，而是让 LLM 给池子里每条启发式打分，依据是当前任务描述的相似度、经验多样性、以及指南本身的信息量。然后只把最相关的 top-k 条注入系统提示。这样既避免了上下文爆炸，又保证了指导的任务相关性。
</div>
</section>

<figure>

<h2>实验设置：Gaia2</h2>

实验是在 Gaia2 上做的。Gaia2 是一个模拟移动环境的 benchmark，包含 12 个应用和 101 个工具。作者选了 Search 和 Execution 两个 split。Search 任务偏重信息检索，Execution 任务偏重多步操作。Gaia2 的一个关键设计是 universe：每个 universe 有相同的应用和工具，但数据完全隔离，比如联系人、文件、日历事件都不一样。作者用 8 个 universe 积累启发式，在 2 个完全陌生的 test universe 上评估。这很像真实场景：工具固定，但数据持续变化。
</div>
</section>

<h2>主结果：56.1% 整体成功率</h2>

结果很直接。ERL 整体成功率 56.1%，比 ReAct 基线高出 7.8 个百分点，也比之前最好的 ExpeL 高出 5.2 个百分点。更值得注意的是，ERL 在 Execution 和 Search 两边都有稳定提升，而 AutoGuide 虽然 Search 很强，Execution 却低于基线，说明它的上下文匹配机制在执行型任务上并不鲁棒。
</div>
</section>

<figure>

<p class="visual-note-index">010 / Video Notes</p>
<h2>启发式 vs 原始轨迹</h2>

论文还做了几个很有信息量的消融实验。第一个对比是启发式 versus 原始轨迹。直接把过去轨迹当成 few-shot 示例塞进提示，结果比基线还低了 1.9%。这说明 raw trajectory 缺少可迁移的抽象，Agent 没法从中提炼出该怎么做。而启发式是蒸馏后的策略原则，上下文效率更高，而且在控制 token 预算的前提下依然占优。
</div>
</section>

<figure>
<img src="{{ '/assets/posts/video-notes/2026-06-25-erl/images/11.png' | relative_url }}" alt="检索质量比数量更重要" loading="lazy">
</figure>
<div markdown="1">
<p class="visual-note-index">011 / Video Notes</p>
<h2>检索质量比数量更重要</h2>

第二个对比是检索机制。如果把 LLM-based 检索换成随机选，性能呈非单调变化，在 40 到 60 条时达到峰值，之后反而下降。这说明经验不是越多越好，关键是选出真正相关的。换成 embedding 检索，整体也只有 53.3%，低于 LLM 检索的 56.1%。
</div>
</section>

<figure>

</figure>
<div markdown="1">
<p class="visual-note-index">012 / Video Notes</p>
<h2>失败 vs 成功：各有最佳战场</h2>

第三个发现是关于失败和成功的价值。只从失败轨迹提取启发式，Search 任务表现极好，比基线高出 14.3%；只从成功轨迹提取，Execution 任务高出 9.0%。原因是失败经验提供了负面约束，能帮 Agent 剪掉无效策略，这在信息检索类任务里特别有用；成功经验则强化了被验证过的动作序列，对执行型任务更直接。当然，在实际部署中任务分布未知，混合检索是最稳妥的折中。
</div>
</section>

<h2>结论：一次教训，长期受益</h2>

最后一个数字是关于可靠性。ERL 在 pass-hat-3 上提升了 8.3% 和 10.6%，意思是 Agent 连续三次都成功的概率显著提高。这说明 ERL 不只是偶尔碰到正确答案，而是让 Agent 的表现更稳定、更可预期。所以这篇论文的真正贡献，不是提出一个更复杂的 Agent 架构，而是指出了一个被忽视的瓶颈：Agent 需要一种轻量的、参数自由的、能从单次经验里持续自我改进的机制。ERL 用反思生成启发式加精准检索注入这两步，把这个机制做轻了、做实了。如果 Agent 部署在真实环境里，任务不能重试、上下文有限、数据持续变化，那 ERL 这种一次教训长期受益的思路，可能比堆砌更多模型参数更值钱。
</div>
</section>
