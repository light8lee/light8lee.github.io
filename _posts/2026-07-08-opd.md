# On Policy Distillaion(OPD)

### 一、On-Policy Distillation 到底是什么

**On-Policy Distillation** 是一种知识蒸馏范式，最核心的原则是：

> 用来训练学生模型的数据，必须由 **学生模型自己当前的策略** 实时生成，而不是来自一个固定的离线数据集。

同时，教师模型（大模型）会实时对这些“自生成”的样本进行打分、修正或提供目标分布，引导学生模型进步。

与之对比：

- **Off-Policy Distillation**：先将教师模型产生的一大堆回答存下来，再用这些静态数据训练学生。学生永远在“模仿历史”，看不到自己犯错后生成的东西。
    
- **On-Policy Distillation**：学生亲自动手试，教师当场批改，学生从自己的错误中学习修正。
    

---

### 二、为什么 LLM 需要 On-Policy Distillation

传统 off-policy 蒸馏存在两个致命问题：

1. **分布偏移（Distribution Shift）**  
    学生模型在学习过程中自身能力会变化，它实际会生成的文本分布和当初教师生成的固定数据分布越来越不一样。用旧数据训练，会越训越偏。
    
2. **曝光偏差（Exposure Bias）**  
    训练时学生看到的都是教师生成的“完美”前缀，但推理时却得靠自己生成的前缀接着写。一步错，步步错，错误会累积。
    

On-policy 的思路让**学生用自己的生成结果作为训练输入**，教师再指导如何纠正，相当于把学生暴露在他可能犯错的真实环境中，从而显著缓解以上问题。

---

### 三、算法流程详解

一个典型的 On-Policy Distillation 循环包含四步：

text

对于一批提示词(prompts)：
  1. 采样（Sample）
     学生模型为每个 prompt 生成 K 条完整回答。
     此时用的是学生当前策略 π_student，包含随机采样（如 temperature > 0）。
  2. 评分（Score）
     教师模型对这些“学生原创回答”进行评估。
     形式灵活，可以是：
     - 给出 token 级的概率分布（软标签，用于计算 KL 散度）
     - 给出整句的质量分数（如奖励模型打分）
     - 给出修正后的文本（像老师改作文）
  3. 构造损失（Loss Construction）
     根据教师反馈构造损失函数，通常包含两部分：
     a) 蒸馏损失：让学生分布靠近教师分布
        - 若有点对点概率：可以用反向 KL 散度 KL(π_student || π_teacher)
        - 若只有打分：可以用策略梯度（类似 PPO），最大化教师给出的奖励
     b) 额外正则：比如熵奖励（鼓励探索）、KL 惩罚（别跑太远）
  4. 更新学生（Update）
     用构造好的损失和自生成的 on-policy 数据做一次或几步梯度下降。
     然后回到步骤 1，用更新后的学生继续采样。

这个循环会重复多轮，学生会越来越强，同时因为它一直在用自己的输出训练，分布始终一致（On-Policy）。

---

### 四、与常见技术的联系与区别

- **和 RLHF（PPO）的关系**  
    如果把教师模型当作奖励模型，上面的步骤 2-3 就和 PPO 很像。  
    实际上，很多现代对齐方案（比如 RLHF with KL to base model）就可以看作一种 **On-Policy Distillation**：学生（当前策略）在线采样，PPO 利用教师信号（奖励模型）来更新。只是这里教师通常是标量奖励，而不是完整概率分布。
    
- **和 DPO 的区别**  
    DPO 是完全**离线**的，用的是提前收集好的偏好对（回答 A 好过回答 B）。  
    On-Policy Distillation 是**在线**的，数据是学生自己一边学一边生成的，教师实时反馈。
    
- **和 Self-Play 微调（如 SPIN）的区别**  
    SPIN 是学生和自己过去的版本对打，没有独立的教师模型。On-Policy Distillation 有明确的、固定的教师模型提供外部知识。
    

---

### 五、在 LLM 中的典型应用场景

1. **对齐与安全训练**  
    用带安全约束的大模型作为教师，让学生在线采样，教师当场判定是否安全并给予纠正信号。比离线数据更灵活，能覆盖学生探索出的各种危险边界。
    
2. **模型压缩**  
    将超大模型蒸馏成小模型时，用 On-Policy 的方式可以大幅提升小模型在复杂推理任务上的表现。比如小模型尝试写推理步骤，大模型实时评判步骤是否合理。
    
3. **工具使用 / 代码生成**  
    学生生成调用工具的代码，教师（可结合解释器）在线给 feedback，修正逻辑错误。这也是 On-Policy Distillation 的一种形式。
    

---

### 六、优缺点总结

**优点**：

- 消除训练-推理分布差异，效果通常远好于离线蒸馏
    
- 学生可以探索并得到教师对自己坏样本的针对性反馈
    
- 与强化学习自然融合，扩展性强
    

**挑战**：

- 计算开销大：每轮都要用学生采样 + 教师推理
    
- 训练不稳定：学生策略变化太剧烈可能导致崩溃，需要仔细设计 KL 约束
    
- 教师模型若本身有偏见，在线反馈可能放大错误

# SDPO和OPD的区别

**OPD 是“学生用自己的 rollout，让外部老师来教”；SDPO 是“模型用环境反馈把自己变成老师，再教回自己”。**

|维度|OPD|SDPO|
|---|---|---|
|全称|On-Policy Distillation|Self-Distillation Policy Optimization|
|本质|蒸馏框架/范式|一种具体的自蒸馏 RL 方法|
|teacher 来源|通常是更强/更大的外部 teacher model|当前模型自己，条件是加入反馈后重新预测|
|训练数据|student 自己生成的轨迹|student rollout + 环境反馈/错误信息/成功样本|
|监督信号|teacher 的 token-level logits/概率分布|feedback-conditioned self-teacher 的 token-level 分布|
|是否需要外部老师|通常需要|不需要|
|是否需要 reward model|一般不强调 reward model|不需要显式 reward model|
|主要解决|off-policy 蒸馏的分布错配|RLVR 中稀疏奖励导致的 credit assignment 问题|
|典型场景|模型压缩、teacher-student 后训练、推理能力蒸馏|代码、数学、工具调用、可验证任务、有错误反馈的环境|

**OPD 怎么理解？**

传统蒸馏常见做法是：teacher 生成答案，student 模仿。但问题是 student 推理时会走到自己的状态分布里，一旦中间出错，后续状态 teacher 数据里没见过。

OPD 的做法是：

`student 自己生成轨迹 teacher 在 student 的轨迹前缀上给 token-level 监督 student 学 teacher 在这些状态下会怎么继续`

所以它叫 **on-policy**：训练状态来自 student 自己，而不是固定 teacher 数据。verl 文档也把 OPD 描述为在 student policy 采样出来的 states 上，把 teacher 知识蒸馏给 student；slime 文档也强调这是在 student 自己的数据分布上蒸馏。来源：[![](https://verl.readthedocs.io/favicon.ico)verl OPD docs](https://verl.readthedocs.io/en/latest/algo/opd.html)、[![](https://www.mintlify.com/favicon.ico)slime OPD docs](https://www.mintlify.com/THUDM/slime/advanced/on-policy-distillation)。

**SDPO 怎么理解？**

SDPO 更像是 OPD 的一个“自蒸馏 + RLVR”变体。它不找外部大老师，而是让当前模型看到反馈后再预测一次，把这个“带反馈后的自己”当老师。

流程大概是：

`模型生成答案 环境返回反馈，比如 runtime error、judge feedback、测试结果 把原轨迹 + 反馈喂回模型 模型重新判断每个位置更应该怎么生成 把这个反馈条件下的分布蒸馏回原 policy`

SDPO 论文主页说得很清楚：它把 tokenized feedback 转成 dense learning signal，不需要 external teacher 或 explicit reward model；当前模型 conditioned on feedback 后作为 self-teacher。来源：[SDPO project page](https://self-distillation.github.io/SDPO)、[![](https://arxiv.org/favicon.ico)arXiv:2601.20802](https://arxiv.org/abs/2601.20802)。

**最关键区别**

OPD 的老师通常是：

`强 teacher model`

SDPO 的老师是：

`同一个模型 + 反馈上下文`

所以 OPD 更像：

`我做一遍，老师看着我的每一步纠正我`

SDPO 更像：

`我做错后看到了错误信息，于是反思出每一步哪里该改，再把这个反思蒸馏回自己`

**面试回答版**

可以这么说：

> OPD 是一个更广义的 on-policy 蒸馏范式，它让 student 在自己的生成分布上采样，再用 teacher 对这些 student-visited states 提供 token-level 监督，从而缓解传统 off-policy KD 的分布错配。SDPO 则是 OPD 思路在 RLVR/rich feedback 场景下的自蒸馏版本，不依赖外部 teacher，而是把当前模型在环境反馈条件下的预测当作 self-teacher，再蒸馏回 policy。OPD 重点是 teacher-student 蒸馏，SDPO 重点是利用反馈做 dense credit assignment。


## Q&A：teacher的那个Prompt的部分也需要包含student那个output是吧,然后相当于output是会用两部分,一个是prompt里,然后再加那个feedback,就是Prompt的部分,然后output是还会再继续用来计算那个loss。

对，但这里要分清楚两个“用 student output 的位置”，否则很容易绕晕。

**1. teacher prompt 里可以包含完整 student output**

这是为了让 self-teacher 知道：

`你刚才完整回答了什么 环境反馈为什么说你错`

比如 teacher 的上下文可以是：

`原始问题： 写 square(x) 模型刚才的完整回答： def square(x): return x + x 环境反馈： square(3)=6，期望 9，应该用乘法。`

这部分属于 **teacher context / reprompt**。

**2. 计算 loss 时，还会再用同一条 output 做 teacher-forcing**

也就是在每个 token 位置上，比较：

`student: P(next_token | 原始问题 + output_prefix) self-teacher: P(next_token | 原始问题 + 完整output + feedback + output_prefix)`

听起来像 output 出现了两次，是的，很多实现概念上就是这样：

`teacher 输入 = [原始 prompt] + [完整 student output] + [feedback] + [用于打分的 output prefix]`

完整 output 用于“反思上下文”，output prefix 用于“对每个 token 做 next-token 分布打分”。

不过实现里不一定真的拼成这么朴素的重复文本，可能会用特殊模板、分隔符、mask、position 处理来避免混淆。核心不变：

`完整 output：让 teacher 看到整次尝试和反馈 output prefix：用于 teacher-forcing 计算每一步 logits`

拿你的话改成更准确的版本：

> teacher 的 prompt 部分通常会包含原始 prompt、student 的完整 output，以及 feedback；然后在算 logits/loss 时，还会把 student output 作为被 teacher-forcing 的 response 序列，对每个 response token 位置计算 next-token distribution，再和 student 分支的 distribution 算 KL。

一个小例子：

`原始 prompt: 写 square(x) student output: def square(x): return x + x feedback: 测试失败，square(3)=6，期望 9。`

student 分支打分：

`输入前缀: 写 square(x) + "def square(x): return x" 预测下一个 token: "+"`

teacher 分支打分：

`反思上下文: 写 square(x) 刚才回答: def square(x): return x + x 反馈: 测试失败，应该用乘法 打分前缀: def square(x): return x 预测下一个 token: teacher 更偏向 "*"`

所以，是你说的那个意思：**student output 一方面作为 teacher prompt 里的“待反思对象”，另一方面作为被计算 loss 的 trajectory。**



# 二者是否是RL
OPD 本身不是严格意义上的 RL，它更像 on-policy knowledge distillation。它让 student 在自己的分布上采样轨迹，再由 teacher 对这些 student-visited states 提供 token-level 监督，训练目标通常是 KL/CE，而不是 reward maximization。SDPO 则更接近 RL，因为它从当前 policy rollout 出发，接收环境 feedback/reward，再把 rich feedback 转成 feedback-conditioned self-teacher 的 dense token-level signal 来更新 policy。因此 SDPO 可以看作 RLVR/RLRF 下的 self-distillation policy optimization，而不是普通 SFT。
